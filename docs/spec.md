# spec

言葉（テキスト・歌詞・台詞）を入力すると、手話/指文字/身振りの3Dモーションを生成し、**任意の3Dアバターの手・指・腕へリターゲット（流し込み）できる**モーション基盤ライブラリの仕様。

コード構成・レイヤ分離・技術スタックは `docs/architecture.md` を参照。ここに書くのはコードが説明できない仕様（データモデル・モードの線引き・retarget 指示形式・MVP スコープ）。

見た目の設計システム（DESIGN.md）は現時点では未作成。理由は「何ではないか」の項を参照（唯一の視覚要素である右上ワイプは利用側の演出であり、このライブラリの設計システムではない）。デモ/プレイグラウンドUI に着手する段で DESIGN.md を起こす。

## 何ではないか

- 「手話翻訳AI」ではない。手話は各言語の逐語置換ではなく、それ自体が文法・空間参照・表情/視線/間を持つ独立言語であり、完全な自動翻訳を名乗らない。
- 右上ワイプを合成する配信アプリではない。ワイプのデザイン・配置・配信画面への合成は、利用側（配信者・動画制作者）が行う。
- 特定のアバターアプリではない。特定アバターに固定されず、手・指・腕のモーションを配り、任意のアバターへ流し込めるライブラリにする。

ライブラリが担うのは、生成された手話/指文字/身振りモーションを、任意の3Dアバターの骨格へマッピング/リターゲットする層。

## 核

- **入力**: プレーンテキスト / 歌詞 / タイムコード付き歌詞 / 台詞。取り込み形式は SRT / VTT / LRC / プレーンテキストを想定。
- **出力**: アバター非依存の両手3Dモーション（手首・手のひら・指関節、必要に応じて腕・顔・体）。runtime が直接読む motion JSON と、エージェントが読める retarget 指示（後述）の両方。
- **プレビュー**: Three.js 上の簡易手モデル（左右の手）。
- **ライブラリ利用**: 好きな VRM / glTF / Three.js アバターへ手話/指文字/身振りモーションを流し込む。

## データモデル（最小中間表現）

アバターに依存しない中間表現として、以下を定義する。**不変の定義データ（プリセット motion・avatar profile）と、実行時に再生される motion/timing を分けて扱う**（定義とその再生処理を同じ構造に混在させない）。

- **skeleton**: 左右の手・手首・手のひら・指関節の構造定義。
- **pose**: 各関節の回転、手首位置、手の向き（1時点の姿勢）。
- **motion**: pose の時系列。
- **timing**: 開始時刻・終了時刻・ease・hold。
- **semantic label**: 単語/句/歌詞/身振り名などの意味ラベル。

### avatar profile（アバター定義）

「このアバターでは、肩/肘/手首/各指がどのボーンか、可動域はどこまでか、初期姿勢をどう補正するか」を定義する。avatar profile は**アバターごとの不変の定義**であり、実行時 motion とは別に持つ。

- 必須: 左手・右手・指ボーン。
- 任意: 肘・肩・顔。
- 指ボーンが不足するアバターでは gesture mode のみ対応に落とす（フォールバック）。
- 可動域を超える指示は、意味を保つ範囲で腕位置を補正する。

## モードの線引き（手話の正確性を誇張しない）

不正確な手話を「手話対応」と言い切ると逆効果になり得るため、モードを名前で明確に分ける。

- **字幕ワイプモード**: 字幕テキストを元に、かな指文字や簡単な身振りを出す。手話通訳とは呼ばない。
- **指文字ワイプモード**: 固有名詞や短い語を指文字で表示する。
- **監修済み手話ワイプモード**: ネイティブ手話話者が確認した語彙・フレーズだけを使う。この段階から初めて「手話対応」と言える。
- **アバターリターゲット**: 好きな VRM / glTF / Three.js アバターへモーションを流し込む。ワイプ化そのものは利用側の演出。

**規律: 実手話モードと手話風身振りモードを混同させない。** 本物の手話支援として提供する段階では、必ず当事者・ネイティブ手話話者のレビューを前提にする。手話として正確に扱う語彙は、後から監修前提で追加する。

## エージェント向け retarget 指示

アバター自体もエージェントで編集できる時代を前提に、runtime が直接読む motion JSON だけでなく、**エージェントへの指示として使いやすい形式**も吐けるようにする。

- **機械可読**: JSON（runtime が直接読む motion データ）。
- **人間/エージェント可読**: YAML / Markdown（retarget タスクの指示書）。

指示の例（どのボーンを特定し、指が足りなければ gesture-only に落とし、手のひら向きを手首位置より優先する、など）を YAML で表現できる形にする。retarget 指示は「必須/任意ボーン」「フォールバック」「意味を保つための優先順位（例: 正確な手首位置より読み取れる手の形を優先）」を含む。

## MVP

1. Three.js 上に左右の手だけを表示する。
2. 数個の手話/身振りプリセットを JSON で定義する（最初は「ありがとう」「こんにちは」「好き」「食べる」「歌う」等の短い語彙）。
3. テキスト入力からプリセット列へ変換する。
4. 時系列で両手を動かす（timing に沿った再生）。
5. 同じ motion JSON を別のアバター手骨格へ流し込む（retarget の疎通）。

手話として正確に扱う語は、この段階では扱わず、後で監修前提にする。

## 歌詞/振付との接続

歌詞は音だけでなく、時間に沿った意味の流れ。耳で聞く・字幕を見る以外に、手の動きとして空間へ置ける。

- 歌詞を入力し、単語やフレーズを手話/身振りモーションへ対応させる。
- タイムコード（LRC 等）に合わせて両手が動く。
- アバターのダンス振付へ手話的な意味を混ぜる。

手話を「翻訳」だけに閉じず、歌詞の意味を踊りやアバター表現へ流し込む基盤にする。
